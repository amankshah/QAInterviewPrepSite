Testing Fundamentals

Q1-15
1 What is Software Testing?
The process of evaluating and verifying that a software product meets specified requirements and works as intended to deliver business value.
2 What is Manual Testing?
Testing software manually without using any automated tools. Testers execute test cases and observe system behavior.
3 What is the difference between Verification and Validation?
Verification: "Are we building the product right?" (Static process - reviews, inspections)
Validation: "Are we building the right product?" (Dynamic process - testing, UAT)
4 What is Quality Assurance (QA)?
Process-oriented approach to ensure quality is built into the development process. Focuses on preventing defects.
5 What is Quality Control (QC)?
Product-oriented approach that identifies defects in the final product. Focuses on detecting defects through testing.
6 What are the objectives of testing?
- Find defects early
- Ensure product meets requirements
- Build confidence in quality
- Prevent defects from reaching production
- Reduce business risk
7 What are the 7 principles of software testing?
1. Testing shows presence of defects
2. Exhaustive testing is impossible
3. Early testing saves time and money
4. Defects cluster together
5. Pesticide paradox
6. Testing is context dependent
7. Absence of errors fallacy
8 What is Black Box Testing?
Testing the functionality without knowing the internal code structure (Behavioral Testing).
9 What is White Box Testing?
Testing the internal code structure and logic (Structural Testing). Requires programming knowledge.
10 What is Grey Box Testing?
Combination of black box and white box testing. Partial knowledge of internal structure.
11 What is a Test Plan?
A document outlining the testing strategy, objectives, scope, resources, schedule, and deliverables for a testing project.
12 What is a Test Strategy?
High-level document defining the testing approach for the organization or project. Defines types of testing, tools, and resources.
13 What is Test Coverage?
Measure of how much of the application has been tested. Can be measured by requirements coverage, code coverage, etc.
14 What is Traceability Matrix?
Document mapping requirements to test cases. Ensures all requirements are covered and helps track testing progress.
15 What is Test Data?
Input data used to execute test cases. Includes both valid and invalid data to test different scenarios.

SDLC & STLC

Q16-30
16 What is SDLC?
Software Development Life Cycle - the complete process of software development: Planning, Design, Implementation, Testing, Deployment, Maintenance.
17 What is STLC?
Software Testing Life Cycle - phases of testing: Requirements Analysis, Test Planning, Test Case Development, Environment Setup, Test Execution, Test Closure.
18 What are the phases of STLC?
1. Requirement Analysis: Understand requirements
2. Test Planning: Define strategy and resources
3. Test Case Development: Write test cases
4. Environment Setup: Prepare test environment
5. Test Execution: Run test cases
6. Test Closure: Complete metrics and reports
19 What is Waterfall Model?
Sequential development model where each phase must be completed before the next begins. Testing happens after development.
20 What is Agile Methodology?
Iterative development approach with short sprints (1-4 weeks). Testing happens continuously throughout development.
21 What is V-Model?
Extension of waterfall where testing activities are planned parallel to development. Each development phase has a corresponding testing phase.
22 What is Sprint in Agile?
A time-boxed iteration (usually 2 weeks) where a specific set of work is completed and made ready for review.
23 What is Scrum?
An Agile framework with defined roles (Scrum Master, Product Owner, Team), events (Sprint, Daily Standup, Retrospective), and artifacts (Product Backlog, Sprint Backlog).
24 What is User Story?
Short, simple description of a feature from user's perspective. Format: "As a [user], I want [goal] so that [benefit]".
25 What is Acceptance Criteria?
Conditions that must be met for a user story to be considered complete and accepted by the Product Owner.
26 What is Definition of Done (DoD)?
Checklist of criteria that must be met for a work item to be considered complete (code review, unit tests, documentation, etc.).
27 What is Entry Criteria?
Prerequisites that must be met before testing can begin (test environment ready, test data available, build deployed, etc.).
28 What is Exit Criteria?
Conditions that must be met to complete testing (all test cases executed, 95% pass rate, no critical bugs, etc.).
29 What is Regression Testing?
Re-testing existing functionality after code changes to ensure nothing broke. Usually performed after bug fixes or new features.
30 What is Retesting?
Testing a specific defect fix to verify it's resolved. Different from regression testing which tests broader functionality.

Test Cases & Scenarios

Q31-45
31 What is a Test Case?
A set of conditions, inputs, and expected results to verify a specific functionality or requirement.
32 What are the components of a Test Case?
- Test Case ID
- Test Description
- Pre-conditions
- Test Steps
- Test Data
- Expected Result
- Actual Result
- Status (Pass/Fail)
- Priority/Severity
33 What is Test Scenario?
High-level functionality to be tested. A scenario can have multiple test cases. Example: "Login Functionality".
34 What is the difference between Test Case and Test Scenario?
Test Scenario: What to test (high-level)
Test Case: How to test (detailed steps)
35 What is Positive Testing?
Testing with valid inputs to verify system works as expected under normal conditions.
36 What is Negative Testing?
Testing with invalid inputs to verify system handles errors gracefully and doesn't crash.
37 What is Boundary Value Analysis?
Testing at the boundaries of input ranges. For range 1-100, test: 0, 1, 100, 101.
38 What is Equivalence Partitioning?
Dividing input data into partitions that should be treated similarly. Test one value from each partition.
39 What is Decision Table Testing?
Test technique using tables to represent combinations of inputs and their corresponding outputs.
40 What is State Transition Testing?
Testing how system behaves when transitioning between different states. Example: Order states (Placed ‚Üí Processing ‚Üí Shipped ‚Üí Delivered).
41 What is Use Case Testing?
Testing based on use cases that describe user interactions with the system to accomplish goals.
42 What is Exploratory Testing?
Simultaneous learning, test design, and execution. Testers explore the application without predefined test cases.
43 What is Ad-hoc Testing?
Informal testing without planning or documentation. Random testing to find defects.
44 What is Monkey Testing?
Random testing by providing random inputs to see if system crashes. No specific test cases.
45 What makes a good Test Case?
- Clear and concise
- Repeatable
- Independent
- Has clear expected results
- Covers one scenario
- Traceable to requirements

Defect Management

Q46-60
46 What is a Defect/Bug?
A flaw in the system that causes it to produce incorrect results or behave unexpectedly.
47 What is Defect Life Cycle?
New ‚Üí Assigned ‚Üí Open ‚Üí Fixed ‚Üí Retest ‚Üí Verified ‚Üí Closed
(Alternative paths: Rejected, Deferred, Reopened, Duplicate)
48 What is Severity?
The technical impact of the defect on the system. Set by QA. Levels: Critical, High, Medium, Low.
49 What is Priority?
The business urgency of fixing the defect. Set by Product Owner. Levels: Immediate, High, Medium, Low.
50 What's the difference between Severity and Priority?
Severity: Technical impact (Application POV)
Priority: Business impact (User/Business POV)
Example: Typo on payment page = Low Severity, High Priority
51 What is a Critical Bug?
A defect that causes system crash, data loss, or makes the application completely unusable.
52 What information should a Bug Report contain?
- Bug ID
- Title/Summary
- Steps to reproduce
- Expected vs Actual result
- Severity & Priority
- Environment details
- Screenshots/Videos
- Logs
53 What is a Blocker Bug?
A defect that blocks further testing or prevents key functionality from working.
54 What is Bug Triage?
Meeting to review, prioritize, and assign bugs. Determines which bugs to fix first.
55 What is a Duplicate Bug?
A bug that has already been reported. Marked as duplicate and linked to the original bug.
56 What is a Deferred Bug?
A bug postponed to a future release due to low priority or time constraints.
57 What is Bug Leakage?
Bugs found by end-users in production that should have been caught during testing.
58 What is Bug Release?
Intentionally shipping a product with known low-priority bugs due to time constraints or business decisions.
59 What is Root Cause Analysis?
Investigating the underlying reason for a defect to prevent similar issues in the future.
60 How do you handle a developer saying "It's not a bug"?
1. Re-verify and replicate
2. Document clearly (video/logs)
3. Cite specific requirement
4. Escalate to PO/Lead if needed

Testing Types & Levels

Q61-80
61 What is Unit Testing?
Testing individual components/functions in isolation. Usually done by developers.
62 What is Integration Testing?
Testing interactions between integrated components/modules to ensure they work together.
63 What is System Testing?
Testing the complete, integrated system to verify it meets specified requirements.
64 What is Acceptance Testing (UAT)?
Testing by end-users to verify the system meets business requirements and is ready for production.
65 What is Smoke Testing?
Quick, shallow testing to verify basic functionality works. Determines if build is stable enough for detailed testing.
66 What is Sanity Testing?
Narrow, deep testing of specific functionality after a bug fix or minor change. Subset of regression testing.
67 What is Functional Testing?
Testing features and functions based on requirements. Verifies "what" the system does.
68 What is Non-Functional Testing?
Testing "how" the system performs (Performance, Security, Usability, Reliability, Scalability).
69 What is Performance Testing?
Testing system performance under load (response time, throughput, resource usage).
70 What is Load Testing?
Testing system behavior under expected user load to verify it handles the load gracefully.
71 What is Stress Testing?
Testing beyond normal capacity to find the breaking point and see how system recovers.
72 What is Security Testing?
Testing to find vulnerabilities and ensure data protection (SQL injection, XSS, authentication, authorization).
73 What is Usability Testing?
Testing user-friendliness, ease of use, and user experience.
74 What is Compatibility Testing?
Testing across different browsers, OS, devices, and screen resolutions.
75 What is Localization Testing?
Testing application for a specific locale (language, currency, date format, cultural aspects).
76 What is Globalization Testing?
Testing that application works across multiple locales without code changes.
77 What is Alpha Testing?
Testing by internal team before release to external users. Done at developer's site.
78 What is Beta Testing?
Testing by a limited external audience in a real environment before full release.
79 What is End-to-End Testing?
Testing complete user workflows from start to finish to ensure all integrated components work together.
80 What is Accessibility Testing?
Testing for users with disabilities (screen readers, keyboard navigation, color contrast, WCAG compliance).

Advanced Topics

Q81-90
81 What is Risk-Based Testing?
Prioritizing testing based on risk assessment. High-risk areas get more testing attention.
82 What is Test Automation?
Using tools and scripts to execute tests automatically. Useful for regression, performance, and repetitive tests.
83 When should you automate tests?
- Repetitive test cases
- Regression testing
- Stable features
- Performance/Load testing
- Data-driven tests
84 What is Test Environment?
Setup of hardware, software, network, and data where testing is performed. Should mimic production.
85 What is a Test Oracle?
Source of expected results to compare against actual results (requirements, specs, previous versions).
86 What is Test Metrics?
Quantitative measures to track testing progress: Test cases executed, Pass/Fail rate, Defect density, Test coverage, etc.
87 What is Defect Density?
Number of defects per unit size (LOC, module, functionality). Formula: Defects / Size.
88 What is Test Effectiveness?
Measure of how well testing finds defects. Formula: Defects found in testing / Total defects.
89 What is Shift-Left Testing?
Starting testing early in the SDLC (during requirements, design). Finds defects earlier when they're cheaper to fix.
90 What is Continuous Testing?
Automated testing integrated into CI/CD pipeline. Tests run automatically with every code change.
