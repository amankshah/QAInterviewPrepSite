Problem Solving & Logical Thinking

Q1-15
1 How would you test a pen?
Functional: Writing on different surfaces, ink flow, click mechanism
Non-functional: Durability, ink longevity, temperature resistance
Usability: Grip comfort, weight, design
Boundary: How long it writes, pressure limits
Compatibility: Different paper types, angles
2 How would you test an elevator?
- Buttons functionality (floor, open/close, alarm)
- Weight limits
- Power failure handling
- Emergency features
- Speed and accuracy
- Multiple call requests
- Door sensors
- Display accuracy
3 How would you test an ATM machine?
- Valid/invalid card handling
- PIN validation (correct, incorrect, blocked)
- Withdrawal limits
- Balance inquiry accuracy
- Receipt printing
- Cash dispensing
- Timeout handling
- Card retention scenarios
4 How would you test a coffee machine?
- Different beverage options
- Water level/temperature
- Payment acceptance
- Cup detection
- Refund functionality
- Multiple orders
- Cleaning cycle
- Error handling (out of ingredients)
5 How many test cases would you write for a login page?
Positive: Valid credentials, remember me, auto-login
Negative: Invalid username, invalid password, both invalid, empty fields
Security: SQL injection, XSS, brute force
UI/UX: Tab order, field masks, error messages
Performance: Response time
Typically 30-50 test cases
6 How would you test Google Search?
- Search functionality (text, images, videos)
- Auto-suggestions
- Spelling corrections
- Filters and sorting
- Pagination
- Voice search

7 How do you approach testing when you have limited information?
1. Explore the application as end-user
2. Document questions
3. Create assumption-based test cases
4. Use exploratory testing
5. Test common user flows
6. Get clarification before final sign-off
8 Given a text field that accepts 1-100, what test cases would you write?
Valid: 1, 50, 100
Boundary: 0, 1, 100, 101
Invalid: -1, 101, abc, special chars
Edge: Empty, spaces, decimal, very large number
Format: Leading zeros (01, 001)
9 How would you test a date picker?
- Date selection (click, type)
- Format display
- Navigation (month, year)
- Min/max date restrictions
- Leap years
- Time zones
- Disabled dates
- Keyboard navigation
- Mobile touch
10 How do you identify test scenarios from requirements?
1. Read requirements thoroughly
2. Identify actors/users
3. List all actions/functions
4. Map inputs and expected outputs
5. Consider edge cases
6. Include error scenarios
7. Check integration points
11 How would you prioritize 100 test cases when you can only run 20?
- Business-critical flows first
- High-risk areas
- Frequently used features
- Recently changed code
- Customer-facing features

12 What's your approach to finding the root cause of a bug?
1. Reproduce consistently
2. Isolate variables
3. Check logs/console
4. Compare with working scenario
5. Test related functionality
6. Check recent changes
7. Collaborate with developers
13 How do you decide when a feature is ready for release?
- All test cases executed
- No critical/high bugs
- Exit criteria met
- Regression passed
- Performance acceptable
- Stakeholder sign-off
- Known issues documented
14 A bug cannot be reproduced - what do you do?
- Document exact steps from reporter
- Try different environments
- Check browser/device variations
- Review logs at time of occurrence
- Mark as intermittent with details
- Set up monitoring/alerts
- Keep the bug open with notes
15 How would you test a website that's completely new to you?
1. Understand purpose and users
2. Explore main navigation
3. Test primary user journeys
4. Check forms and inputs
5. Test across browsers/devices
6. Look at error handling
7. Check performance
8. Review accessibility

Technical Concepts

Q16-30
16 Explain the difference between verification and validation.
Verification: "Are we building the product right?" - Reviews, walkthroughs, inspections (static testing)
Validation: "Are we building the right product?" - Testing the actual product (dynamic testing)
17 What is the difference between a bug, defect, and error?
Error: Human mistake in code or understanding
Defect: Variance from requirement (bug in code)
Bug: Informal term for defect, often found in testing
Failure: Defect manifesting during execution
18 What is boundary value analysis?
Testing technique focusing on values at boundaries. For range 1-100: test 0, 1, 2, 99, 100, 101. Bugs often occur at boundary edges.
19 What is equivalence partitioning?
Dividing input data into valid/invalid partitions where all values in a partition behave similarly. For age 1-100: partitions are <1, 1-100, >100. Test one value from each partition.
20 What is regression testing?
Re-running existing tests after code changes to ensure nothing is broken. Focuses on affected areas plus critical paths. Prime candidate for automation.
21 What is smoke testing vs sanity testing?
Smoke: Broad, shallow testing of critical features. "Does the build work at all?" Done after new build.
Sanity: Narrow, deep testing of specific functionality. "Is this fix working?" Done after minor changes.
22 What is alpha testing vs beta testing?
Alpha: Internal testing by organization's team in dev environment
Beta: External testing by real users in production-like environment before release
23 What is load testing vs stress testing vs performance testing?
Performance: Umbrella term for speed/responsiveness tests
Load: Testing under expected load conditions
Stress: Testing beyond normal capacity to find breaking point
24 What is a test case vs test scenario?
Test Scenario: High-level description (what to test)
Example: "Test user login"
Test Case: Detailed steps, data, expected result (how to test)
Example: Steps to login with valid credentials
25 What is decision table testing?
Technique using tables to represent combinations of inputs and expected outputs. Useful when multiple conditions affect behavior. Ensures all combinations are tested.
26 What is state transition testing?
Testing valid/invalid state changes in a system. Example: Order states (Created ‚Üí Paid ‚Üí Shipped ‚Üí Delivered). Tests transitions between states and invalid transition attempts.
27 What is test coverage? What types exist?
Measure of how much code/requirements are tested:
- Code coverage: Lines, branches, conditions
- Requirement coverage: Tests per requirement
- Risk coverage: Coverage of risky areas
28 What is a traceability matrix?
Document mapping requirements to test cases. Shows coverage, identifies gaps, tracks changes. Bidirectional: requirement ‚Üí test cases, test case ‚Üí requirements.
29 What is mutation testing?
Technique to evaluate test quality by introducing small code changes (mutants). Good tests should fail when mutants are introduced. Measures test effectiveness.
30 What is pair testing?
Two testers work together on same testing tasks. One drives, one observes. Combines different perspectives, catches more bugs, knowledge sharing.

Process & Methodology

Q31-45
31 Describe your typical day as a QA engineer.
- Check build status and test results
- Daily standup meeting
- Execute test cases or automation
- Review requirements/user stories
- Log and triage bugs
- Collaborate with developers
- Update test documentation
- Prepare for next sprint
32 How do you write a good bug report?
Essential elements:
- Clear, descriptive title
- Environment details
- Steps to reproduce
- Expected vs actual results
- Severity/priority
- Screenshots/videos
- Logs if applicable
33 What's the difference between Agile and Waterfall for testing?
Waterfall: Sequential, testing after development, big-bang integration
Agile: Iterative, testing throughout, continuous integration, collaboration-focused
34 What is TDD (Test-Driven Development)?
Development approach: Write test first ‚Üí Run (fails) ‚Üí Write code ‚Üí Run (passes) ‚Üí Refactor. Tests drive the design. Results in better test coverage.
35 What is BDD (Behavior-Driven Development)?
Extension of TDD focusing on behavior. Uses plain English (Gherkin): Given-When-Then. Bridges communication between business and technical teams. Tools: Cucumber, SpecFlow.
36 What are entry and exit criteria for testing?
Entry: Conditions to start testing
- Requirements approved
- Test environment ready
- Build deployed
Exit: Conditions to end testing
- Test cases executed
- Bugs fixed/documented
- Coverage achieved
37 What is CI/CD and how does testing fit in?
CI (Continuous Integration): Frequent code merges, automated builds and tests
CD (Continuous Delivery/Deployment): Automated release pipeline
Testing: Unit, integration, smoke tests run automatically in pipeline
38 What is a test plan and what does it contain?
Document outlining test approach:
- Scope and objectives
- Test strategy
- Test environment
- Resources and schedule
- Entry/exit criteria
- Deliverables
- Risks and mitigations
39 How do you handle testing in a sprint?
- Review stories during planning
- Write test cases early
- Test features as completed
- Automation alongside manual
- Daily collaboration with devs
- Bug fixing within sprint
- Demo tested features
40 What is shift-left testing?
Moving testing earlier in SDLC. Review requirements, participate in design, write tests before code. Catches defects early when they're cheaper to fix.
41 How do you measure testing effectiveness?
- Defect detection rate
- Defect leakage to production
- Test coverage percentage
- Automation rate
- Test execution time
- Cost of quality metrics
42 What is risk-based testing?
Prioritizing testing based on risk analysis. Identify risks (impact √ó probability), test high-risk areas first. Ensures critical areas get adequate attention.
43 What is defect density and how is it calculated?
Number of defects per unit size of code.
Defect Density = Defects / Lines of Code (or Function Points)
Lower is better. Used to compare quality across modules.
44 What is test closure activities?
- Test completion report
- Archive test artifacts
- Lessons learned meeting
- Release notes contribution
- Update test documentation
- Handover to operations
45 What is the role of QA in Scrum ceremonies?
Planning: Estimate testing, identify risks
Daily: Progress, blockers
Review: Demo tested features
Retro: Process improvements
Refinement: Clarify requirements

Tools & Technologies

Q46-60
46 What automation tools have you worked with?
UI: Selenium, Playwright, Cypress, WebDriverIO
API: Postman, RestAssured, Karate
Mobile: Appium, XCUITest, Espresso
Performance: JMeter, Gatling, k6
47 What test management tools do you know?
- TestRail
- Zephyr
- qTest
- Azure DevOps Test Plans
- Xray for Jira
- PractiTest
48 How do you choose between different automation tools?
- Application type (web, mobile, desktop)
- Team skills
- Language preferences
- Browser/platform support
- Community and documentation
- Integration capabilities
- Cost/licensing
49 What is Page Object Model?
Design pattern for test automation. Each page/component has its own class containing locators and methods. Benefits: reusability, maintainability, cleaner test code.
50 What is data-driven testing?
Running same test with multiple data sets from external sources (Excel, CSV, database). Separates test logic from data. Good for testing many input combinations.
51 What is keyword-driven testing?
Framework where test actions are defined as keywords. Non-technical users can create tests using keywords. Example keywords: Click, Type, VerifyText. Enables collaboration.
52 How do you handle test data?
- Separate data files (JSON, Excel)
- Database seeding
- API calls for setup
- Faker libraries for generation
- Environment-specific configs
- Data cleanup strategies
53 How do you debug failing tests?
- Review error messages/screenshots
- Check test logs
- Run locally with debug mode
- Verify locators still valid
- Check test data
- Review recent code changes
- Check environment status
54 What is flaky test and how do you handle it?
Test that sometimes passes, sometimes fails. Causes: timing issues, shared state, environment. Solutions: proper waits, test isolation, stable locators, retry mechanisms, investigation and fix.
55 How do you structure your automation framework?
- Page objects/components
- Test data management
- Configuration files
- Utility/helper functions
- Test suites organization
- Reporting setup
- CI/CD integration
56 What CI/CD tools have you used?
- Jenkins
- GitHub Actions
- GitLab CI
- Azure DevOps
- CircleCI
- TeamCity
57 How do you integrate tests with CI/CD?
- Configure test job in pipeline
- Set up test environment
- Run tests on trigger (push, PR)
- Generate reports/artifacts
- Fail pipeline on test failures
- Notifications for results
58 What is Docker and how is it useful for testing?
Containerization platform. Benefits for testing:
- Consistent environments
- Easy setup/teardown
- Isolation
- Scalable test execution
- Run dependencies (DB, services)
59 How do you handle cross-browser testing?
- Define browser matrix
- Selenium Grid
- Cloud platforms (BrowserStack, Sauce Labs)
- Parallel execution
- Focus on major browsers
- Screenshot comparison
60 What version control practices do you follow for tests?
- Tests in same repo as code
- Branch per feature
- Code reviews for tests
- Meaningful commit messages
- No hardcoded values
- Keep tests updated with code
